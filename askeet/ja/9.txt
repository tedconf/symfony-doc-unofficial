==================================================

復習
----

[8日目](8.txt)の作業でAJAXインタラクションは、askeetに苦もなく採り入れられました。現在アプリケーションは問題なく使えるものですが、多くにまだ改善の余地があります。質問の<code>`<body>`</code>にはリッチテキストの入力が許可されるべきですし、長いリストはページ分割されなければなりません。そして、プライマリキーはURIに現れてはなりません。そのようなものすべては、symfonyで簡単に実現できます。今日は、あなたがすでに学んだものを使って、あなたがMVCアーキテクチャーの全ての層を操る方法を知っていることを確認するのに良い機会です。

質問と答えにリッチテキストを使えるようにする
--------------------------------------------

### Markdown

今は質問と答えの欄はプレーンテキストのみが入力できます。基本的なフォーマット（太字やイタリック体、リンク、画像など）を使うために車輪の再開発をするのではなく、すでにある外部ライブラリを使います。 

今は質問と答えの欄はプレーンテキストのみが入力できます。基本的なフォーマット（太字やイタリック体、リンク、画像など）を使うために車輪の再開発をするのではなく、すでにある外部ライブラリを使います。symfonyのドキュメントをテキストフォーマットで見た人はわかると思いますが、私たちは[Markdown](http://daringfireball.net/projects/markdown/)の大ファンなのです。Markdownはテキスト形式をHTML形式に変換するツールです。Wikiなどの文法に比べてMarkdownのプレーンテキストはとても読みやすいことが大きなアドバンテージです:

    Markdownのテキストをテストする
    ------------------------------
    
    これは[Markdown][1]の**とてもシンプルな**例です。
    markdownに関するベストな機能はコードの塊を_オートエスケープ_することです:
    
        <a href="http://www.symfony-project.com">link to symfony</a>
    
    >`<`と`>`は適切に`&lt;`と`&gt;`にエスケープされ、
    >ブラウザによって解釈されません
    
    [1]: http://daringfireball.net/projects/markdown/   "Markdown"

このMarkdownは以下のように表示されます。

>Markdownのテキストをテストする
>------------------------------
>
>これは[Markdown][1]の**とてもシンプルな**例です。
>markdownに関するベストな機能はコードの塊を_オートエスケープ_することです:
>
>      <a href="http://www.symfony-project.com">link to symfony</a>
>
>>`<`と`>`は適切に`&lt;`と`&gt;`にエスケープされ、

### Markdownのライブラリ

Markdownのオリジナルは[Perl](http://www.perl.com/)で書かれたものでしたが、[PHP Markdown](http://www.michelf.com/projects/php-markdown/)にPHPのライブラリ版もあります。これが今から使うものです。`markdown.php`ファイルをダウンロードしてaskeetのプロジェクトディレクトリ(`askeet/lib/`)の`lib`フォルダに保存してください。これで完了です。askeetの全てのクラスから以下のように記述することで使えるようになりました:

    [php]
    require_once('markdown.php');   

本文を表示するたびにMarkdownで変換することもできますが、サーバの負荷になってしまいます。むしろ、質問が作られたときにテキストからHTMLに変換して`Question`テーブルに保存したほうがいいでしょう。そろそろ慣れてきたのでmodelを拡張するのにも驚かなくなってきたでしょう。

### モデルの拡張

まず、`schema.xml`ファイルの`Question`テーブルにカラムを追加します:

    [php]
    <column name="html_body" type="longvarchar" />

そして、モデルをリビルドして、データベースを更新します:

    $ symfony propel-build-model
    $ symfony propel-build-sql
    $ symfony propel-insert-sql

### setBodyメソッドのオーバーライド 

`Question`クラスの`setBodyメソッド`が呼ばれたとき、テキストボディのMarkdown規約に従って`html_body`を更新しなければなりません。`askeet/lib/model/Question.php`ファイルを開き以下のコードを作成してください:

    [php]
    public function setBody($v)
    {
      parent::setBody($v);
  
      require_once('markdown.php');
      
      // HTMLタグをストリップする
      $v = htmlentities($v, ENT_QUOTES, 'UTF-8');
            
      $this->setHtmlBody(markdown($v));
    }

`HtmlBody`をセットする前に`htmlentities()`関数を適用するのはクロスサイトスクリプティング(XSS)から守るためです。そうすることで全ての`<script>`タグはエスケープされます。

### テンプレートを修正する

`question`モジュールの`showSuccess.php`テンプレートを少し修正できます:

    [php]
    ...
    <div class="question_body">
      <?php echo $question->getHtmlBody() ?>
    </div>
    ...
    
listテンプレートフラグメント(`_list.php`)はbodyも表示しますが、省略バージョンです:

    [php]
    <div class="question_body">
      <?php echo truncate_text(strip_tags($question->getHtmlBody()), 200) ?>
    </div>

最後のテストのための準備はすべて整いました: 修正された3つのページを表示し、テストデータから整形されたテキストを見ることになります:

    http://askeet/question/list
    http://askeet/recent
    http://askeet/question/show/stripped_title/what-shall-i-do-tonight-with-my-girlfriend    

![markdownのテキスト](/images/askeet/markdown_question.gif)

`Answer`の`body`にも同じことが当てはまります: `html_body`カラムはモデルの中で作成されなければなりません。`->setBody()`メソッドは上書きされることを必要とし、`question/show`に表示された回答は`->getBody()`メソッドの代わりに`->getHtmlBody()`メソッドを使わなければなりません。コードが上記の物とまったく同じなので、ここでは記述しないことにします。しかし、今日のSVNコードの中で見つかります。

すべてのidを隠す
------------------

symfonyアクションにおける他のグッドプラクティスはリクエストパラメータとしてのプライマリキーをできる限り隠して渡すことです。なぜならプライマリキーは主にオートインクリメントで、データベースのレコードについて多すぎる情報をハッカーに与えてしまうからです。加えて、表示されたURIでは何も意味せず、サーチエンジンに対してよくないからです。

ユーザのプロファイルページを例に考えましょう。現時点ではユーザ`id`はパラメータとして利用されています。しかし、`nickname`が唯一であることを確認したとすれば、リクエストのためのパラメータも同じであったはずです。やってみましょう。

### アクションを変更する

`user/show` アクションを編集して下さい:

    [php]
    public function executeShow()
    {
      $this->subscriber = UserPeer::retrieveByNickname($this->getRequestParameter('nickname'));
      $this->forward404Unless($this->subscriber);
    
      $this->interests = $this->subscriber->getInterestsJoinQuestion();
      $this->answers   = $this->subscriber->getAnswersJoinQuestion();
      $this->questions = $this->subscriber->getQuestions();
    }

### モデルを変更する

次のメソッドを`askeet/lib/model/` ディレクトリにある`UserPeer`クラスに追加して下さい。

    [php]
    public static function retrieveByNickname($nickname)
    {
      $c = new Criteria();
      $c->add(self::NICKNAME, $nickname);

      return self::doSelectOne($c);
    }
 
### テンプレートを変更する

ユーザのプロファイルへのリンクを表示するページは`id`の代わりにユーザの`nickname`を記載しなければなりません。

`question/showSuccess.php`、`question/_list.php`テンプレートにおいて:

    [php]
    <?php echo link_to($question->getUser(), 'user/show?id='.$question->getUserId()) ?>

次のコードで置き換えて下さい:

    [php]
    <?php echo link_to($question->getUser(), 'user/show?nickname='.$question->getUser()->getNickname()) ?>
    
`answer/_answer.php`テンプレートにも同じような修正を行います。

### ルーティングルールの追加

`url`が`nickname`リクエストパラメータを表示するようにこのアクションに対するルーティング構成に新しいルールを追加して下さい:

    user_profile:
      url:   /user/:nickname
      param: { module: user, action: show }    

`symfony clear-cache`を行った後、最後にしなければならないのは修正に対するテストです。

ルーティング
------------

今日の追加とは別に、これまでに多くのアクションがデフォルトのルーティングを利用しているので、モジュール名とアクション名はブラウザのアドレスバーにたびたび表示されます。あなたはそれらを修正することを既に学んだので、すべてのアクションのためにURLパターンを定義しましょう。`askeet/apps/frontend/config/routing.yml`を編集して下さい:

    # question
    question:
      url:   /question/:stripped_title
      param: { module: question, action: show }

    popular_questions:
      url:   /index/:page
      param: { module: question, action: list, page: 1 }

    recent_questions:
      url:   /recent/:page
      param: { module: question, action: recent, page: 1 }

    add_question:
      url:   /add_question
      param: { module: question, action: add }

    # answer
    recent_answers:
      url:   /recent/answers/:page
      param: { module: answer, action: recent, page: 1 }

    # user
    login:
      url:   /login
      param: { module: user, action: login }

    logout:
      url:   /logout
      param: { module: user, action: logout }

    user_profile:
      url:   /user/:nickname
      param: { module: user, action: show }

    # default rules
    homepage:
      url:   /
      param: { module: question, action: list }

    default_symfony:
      url:   /symfony/:action/*
      param: { module: default }

    default_index:
      url:   /:module
      param: { action: index }

    default:
      url:   /:module/:action/*

製品環境を操縦するとしたら、このコンフィギュレーション修正のテストを行う前にキャッシュをクリアすることを強くお勧めします。

symfonyルーティングのグッドプラクティスの1つは`module/action`の代わりに`link_to()`ヘルパにおいて規則名を使用することです。動作を速くするだけでなく(ルーティングエンジンは規則の適応を見つけるためのルーティングコンフィギュレーションの解析を行う必要がないです)、背後にある規則名に対してアクションを修正することもできるようになります。symfony bookの[ルーティングの章](http://www.symfony-project.com/book/1_0/09-Links-and-the-Routing-System)でもっと詳細なことが書かれています。

    [php]
    <?php link_to('@user_profile?id='.$user->getId()) ?>
    // 下記のコードよりもベター
    <?php link_to('user/show?id='.$user->getId()) ?>
 
askeetはsymfonyのグッドプラクティスに従っているので、今日のチュートリアルの終了時にダウンロードするであろうコードにはリンクヘルパの規則名だけ含まれています。すべてのテンプレートとカスタムヘルパにおいて`@rule`によって`action/module`を置き換えることはあまり面白くないことなので、ルーティングに関する最後のアドバイスは次のようにします: アクションを作成することと同じようにルーティングルールを書き、始めからリンクヘルパにおいて規則名を使用して下さい。 

それではまた明日
----------------

今日の変更内容は理解することよりも読むことの方が長かったです。加えて、チュートリアルで記述された修正は全体的なコードにおいて似たようなケースのために繰り返されました。今日は新しい機能は追加されませんでしたが、コードの変更をたくさん行いました。

今日のチュートリアルであまりたくさん勉強したと感じられなかったら、あなた自身のプロジェクトを始める準備が整いつつあることを意味します。アクション作成のプロセス、必要なアクションを提供するためのモデルを変更すること、アクションを出力するためにシンプルなテンプレートを書き、新しいアクションをアプリケーションのロジックを統合するためにコンフィギュレーションを編集することはsymfony開発の基礎です。

ここで紹介されたすべてのグッドプラクティス(symfonyのために書き直すことなく外部のライブラリを利用すること、アプリケーションの主キーを表示しないこと、`module/action`の代わりにルーティングルール名を使用すること)によってアプリケーションをすっきりさせ、安全に、動作速度を速くし、持続可能にします。

しかしながら、askeetのアプリケーションは終了とはほど遠いです!もっとも欠けている機能性は新しい質問を追加することと新しい回答を追加することです。そのことについては明日開発することにします。
